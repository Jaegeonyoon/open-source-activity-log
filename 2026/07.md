## 2026-07-09 - OpenTelemetry documentation: Ruby `in_span` exception handling

Type: Documentation PR  
Status: Merged  
Repository: `open-telemetry/opentelemetry.io`  
Link: [PR #10623](https://github.com/open-telemetry/opentelemetry.io/pull/10623)  
Evidence value: High

### Context

The Ruby documentation around `in_span` exception behavior needed clearer wording.

The relevant behavior involved exception recording, span status, and re-raising when an exception escapes the block.

### What I did

- Clarified how Ruby `in_span` handles exceptions that escape the block.
- Explained that the span records the exception by default.
- Clarified that span status is set to `Error` in that case.
- Clarified that the exception is re-raised.
- Added wording around disabling automatic exception recording.
- Added a short example for exceptions rescued inside the block.

### Verification / Evidence

- Public PR: [open-telemetry/opentelemetry.io#10623](https://github.com/open-telemetry/opentelemetry.io/pull/10623)
- Status: merged
- Changed files: 1

### What I learned

- Documentation contributions often require precise behavior-level language, not only readable prose.
- Exception behavior needs special care because recording, status, and control flow are separate concerns.
- Public technical writing should make runtime behavior explicit without overexplaining.

### Next action

No immediate action. Preserve as a merged documentation contribution.

## 2026-07-08 - OpenTelemetry Helm charts: clusterMetrics with multiple replicas

Type: Helm chart behavior PR  
Status: Merged  
Repository: `open-telemetry/opentelemetry-helm-charts`  
Link: [PR #2298](https://github.com/open-telemetry/opentelemetry-helm-charts/pull/2298)  
Evidence value: High

### Context

The `clusterMetrics` preset was blocked with `replicaCount > 1` even when leader election was available.

Leader election exists to make multi-replica behavior safe, so a stricter unconditional block on multiple replicas was not aligned with the intended behavior.

### What I did

- Relaxed the validation so that multiple replicas are allowed when leader election is not explicitly disabled.
- Adjusted render-time failure behavior in `NOTES.txt`.
- Updated leader-election conditions in `_config.tpl`.
- Updated Lease-related RBAC conditions in `clusterrole.yaml`.
- Verified deployment and statefulset render behavior with `replicaCount: 2` and `clusterMetrics.enabled: true`.
- Verified that rendering fails when `disableLeaderElection: true` is used with `replicaCount: 2`.

### Verification / Evidence

- Public PR: [open-telemetry/opentelemetry-helm-charts#2298](https://github.com/open-telemetry/opentelemetry-helm-charts/pull/2298)
- Status: merged
- Listed testing included:
  - `helm lint charts/opentelemetry-collector`
  - `make check-examples CHARTS=opentelemetry-collector`
  - deployment rendering check
  - statefulset rendering check
  - failure case check when leader election is disabled

### What I learned

- Helm chart validation must match actual safety mechanisms, not only conservative defaults.
- Leader election affects both configuration and RBAC requirements.
- Rendering tests are important for chart changes because the user-facing result is generated YAML.

### Next action

No immediate action. Preserve as a merged Helm chart contribution.

## 2026-07-01 - Kubernetes documentation: container memory metrics clarification

Type: Documentation PR  
Status: Open  
Repository: `kubernetes/website`  
Link: [PR #56346](https://github.com/kubernetes/website/pull/56346)  
Evidence value: Medium

### Context

The Kubernetes Resource metrics pipeline documentation explains memory reporting through the Metrics API. The relationship between that value and the broader set of `container_memory_*` metrics exposed by cAdvisor or Prometheus can be misunderstood.

### What I did

- Clarified that the Metrics API memory value is not interchangeable with every `container_memory_*` metric.
- Mentioned examples such as working set, usage, RSS, cache, mapped files, swap, and maximum recorded usage.
- Pointed readers toward cAdvisor Prometheus metric definitions for detailed metric meaning.
- Related the clarification to existing issue context.

### Verification / Evidence

- Public PR: [kubernetes/website#56346](https://github.com/kubernetes/website/pull/56346)
- Status: open
- Changed files: 1

### What I learned

- Metrics documentation must distinguish similarly named values that come from different layers or APIs.
- Kubernetes documentation changes may require careful alignment with issue history and maintainer expectations.
- Open PRs should be recorded with neutral status language until review outcome is known.
