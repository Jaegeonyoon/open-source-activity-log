## 2026-07-13 - Cross-repository design for parsed MCP transport observation

Type: Issue Investigation / Technical Design Discussion  
Status: Under discussion  
Dependency status: Blocked on upstream callback contract  
Repositories discussed:

* `modelcontextprotocol/go-sdk`
* `CaliLuke/loom-mcp`

Supporting codebase analyzed:

* `CaliLuke/loom`

Links:

* [modelcontextprotocol/go-sdk#1076](https://github.com/modelcontextprotocol/go-sdk/issues/1076)
* [CaliLuke/loom-mcp#271](https://github.com/CaliLuke/loom-mcp/issues/271)

Evidence value: Medium  
Publicly usable: Yes — as evidence of issue investigation and technical design discussion

### Context

This investigation connected an upstream SDK callback contract with downstream Loom-MCP transport observation.

The upstream issue concerns exposing parsed JSON-RPC envelope metadata without requiring downstream code to reread the HTTP request body. The downstream issue concerns using that parsed metadata to enrich Loom-MCP transport events while preserving the existing request observer model.

### What I did

* Analyzed where an upstream parsed-envelope hook should fit in the SDK request lifecycle.
* Distinguished parsed JSON-RPC envelope metadata from raw request-body rereading.
* Connected the upstream callback contract to Loom-MCP’s SDK-backed server path.
* Proposed attaching the existing request observer to the HTTP request context.
* Proposed composing with any caller-provided SDK callback rather than replacing it.
* Identified that the downstream implementation should reuse the existing `transport.RequestObserver`.
* Distinguished request-side session IDs from response-side session establishment.
* Identified Loom-side gaps around batch method lists and numeric JSON-RPC ID normalization.
* Avoided proposing local request-body buffering as a workaround.

### Verification / Evidence

* Public issue discussion in `modelcontextprotocol/go-sdk#1076`.
* Public issue discussion in `CaliLuke/loom-mcp#271`.
* Supporting source analysis of Loom’s `transport.Event`, `RequestObserver`, `SetJSONRPC`, `SetSession`, request-context observer propagation, generated SDK handler lifecycle, and `jsonrpc.IDToString`.

### What I learned

* Cross-repository design work requires separating upstream API contracts from downstream implementation details.
* Observability integration should preserve existing request lifecycle timing and avoid duplicating transport parsing.
* Request-context propagation and callback composition solve different parts of the integration problem.
* Session metadata must be handled carefully because response headers alone may not prove successful session initialization.
* Batch JSON-RPC requests need different event semantics from singular requests.

### Next action

Wait for upstream maintainer feedback before opening the SDK-dependent downstream implementation. Loom-side preparatory changes, including batch method metadata and numeric JSON-RPC ID normalization, can be evaluated independently.

## 2026-07-08 - OpenTelemetry Helm charts: clusterMetrics with multiple replicas

Type: Helm chart behavior PR  
Status: Merged  
Repository: `open-telemetry/opentelemetry-helm-charts`  
Link: [PR #2298](https://github.com/open-telemetry/opentelemetry-helm-charts/pull/2298)  
Evidence value: High

### Context

The `clusterMetrics` preset was blocked with `replicaCount > 1` even when leader election was available.

Leader election exists to make multi-replica behavior safe, so a stricter unconditional block on multiple replicas was not aligned with the intended behavior.

### What I did

- Relaxed the validation so that multiple replicas are allowed when leader election is not explicitly disabled.
- Adjusted render-time failure behavior in `NOTES.txt`.
- Updated leader-election conditions in `_config.tpl`.
- Updated Lease-related RBAC conditions in `clusterrole.yaml`.
- Verified deployment and statefulset render behavior with `replicaCount: 2` and `clusterMetrics.enabled: true`.
- Verified that rendering fails when `disableLeaderElection: true` is used with `replicaCount: 2`.

### Verification / Evidence

- Public PR: [open-telemetry/opentelemetry-helm-charts#2298](https://github.com/open-telemetry/opentelemetry-helm-charts/pull/2298)
- Status: merged
- Listed testing included:
  - `helm lint charts/opentelemetry-collector`
  - `make check-examples CHARTS=opentelemetry-collector`
  - deployment rendering check
  - statefulset rendering check
  - failure case check when leader election is disabled

### What I learned

- Helm chart validation must match actual safety mechanisms, not only conservative defaults.
- Leader election affects both configuration and RBAC requirements.
- Rendering tests are important for chart changes because the user-facing result is generated YAML.

### Next action

No immediate action. Preserve as a merged Helm chart contribution.
